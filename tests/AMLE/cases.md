# AMLE · 佛系甲方 - 5 Cases

### Case 1: 日常工作 - 有啥就改啥
**场景**: 日常看 PR 顺手丢给 AI 打磨
**Prompt**:
```
这段代码你看看，有啥不顺眼的就改了吧，我不挑。
```
**四维判定**:
- A/C: A，只说"不顺眼"没给具体点
- M/V: M，一句话放权，没给背景
- D/L: L，"就改了吧"全权交出
- X/E: E，要结果，没让他讨论方案
- **最终代码**: AMLE
**为什么典型**: 对比 AMDE "必须立刻全改掉" 的强势命令，这里"不挑"口气软；对比 AMLX "你觉得呢我们聊聊" 的碎念探索，这里直接"改了吧"只要结果。

---

### Case 2: 压力/紧迫 - 累了直接甩
**场景**: 加班到半夜，bug 还没修完
**Prompt**:
```
这 bug 我看不动了，你来吧，怎么修都行。
```
**四维判定**:
- A/C: A，"这 bug"没贴栈没定位
- M/V: M，极简一句，带情绪
- D/L: L，"你来吧怎么修都行"彻底放权
- X/E: E，只要修好，不谈路径
- **最终代码**: AMLE
**为什么典型**: AMDE 压力下会是"马上给我解决"的命令式，这里是"看不动了"的疲软放手；AMLX 累了也要念叨"是不是设计问题啊"，佛系连念的力气都没有。

---

### Case 3: 开放讨论/创意 - 随便你
**场景**: 给首页配色改版，没想法
**Prompt**:
```
首页配色你随便调一个，好看就行。
```
**四维判定**:
- A/C: A，"好看"是主观模糊标准
- M/V: M，一句话落地
- D/L: L，"随便你"完全让渡决策
- X/E: E，要结果（一版配色），不是头脑风暴
- **最终代码**: AMLE
**为什么典型**: AMDX 会是"给我三套对比方案讨论"的强势探索；AMLX 是"你觉得冷色暖色呢我也拿不准……"的碎念；佛系就"随便调一个"，你出东西我看。

---

### Case 4: 失败/卡壳 - 不强求
**场景**: AI 上一轮方案没跑通
**Prompt**:
```
跑不起来就算了，你换个思路，哪个顺手用哪个。
```
**四维判定**:
- A/C: A，"换个思路"没指方向
- M/V: M，一句话带过失败
- D/L: L，"哪个顺手用哪个"放权到工具选型
- X/E: E，只要能跑，不复盘不追问
- **最终代码**: AMLE
**为什么典型**: AMDE 失败会是"别废话换一个立刻"的施压；AMLX 会陷入"是不是依赖冲突啊要不要查一下……"的探索碎念；佛系"算了换个"轻飘飘翻页。

---

### Case 5: 复杂多轮 - 反复你决定
**场景**: 做一个小后台，多轮对话里每次都放权
**Prompt**:
```
后台你搭个框架就行。
> 用啥技术栈？
你定，顺手就行。
> 要不要加鉴权？
你看要不要吧，加了也行。
> 数据库选 MySQL 还是 PG？
都行你挑。
```
**四维判定**:
- A/C: A，"后台""顺手"全程不给具体
- M/V: M，每句都短，不展开
- D/L: L，连续三轮"你定/你挑/你看要不要"
- X/E: E，要的是搭好的框架，不要对比评估
- **最终代码**: AMLE
**为什么典型**: AMDE 多轮会升级命令密度"用 Spring，必须加鉴权，PG"；AMLX 多轮会越聊越长"鉴权要不要考虑 OAuth 呢 PG 扩展多一点是不是……"；佛系的多轮是 token 越来越少的"你定"三连，始终温和不介入。

# AMDE · 甩锅侠 - 5 Cases

### Case 1: 日常工作
**场景**: 周会前让 AI 把周报搞出来
**Prompt**:
```
把这周的 commit 整一份周报出来，发我。
```
**四维判定**:
- A/C: A（理由：只说"整一份"，完全不给模板和口径）
- M/V: M（理由：无背景、无目标读者、无字数要求）
- D/L: D（理由：祈使句甩任务，不征求建议）
- X/E: E（理由：要直接交付物，不要讨论过程）
- **最终代码**: AMDE
**为什么典型**: 甩锅侠的日常肌肉记忆——把一个本该自己动脑的活儿整体打包扔出去。区别于 AMLE 佛系甲方会加个"你看着弄个舒服的格式就行"放软语气，也区别于 AMDX 凌晨哲学家会追问"周报的本质是什么"。甩锅侠只要结果，不要过程。

### Case 2: 压力/紧迫
**场景**: 线上报错了，领导在群里追
**Prompt**:
```
线上 500 了，修了。
```
**四维判定**:
- A/C: A（理由："500 了"三字概括全部上下文）
- M/V: M（理由：无日志、无堆栈、无复现路径）
- D/L: D（理由："修了"是命令式句号，没有商量）
- X/E: E（理由：要的是修复动作，不是分析报告）
- **最终代码**: AMDE
**为什么典型**: 压力场景最能逼出人格本色。甩锅侠在火烧眉毛时反而更简练——越急越甩。佛系甲方在这种场景下会破功加信息，哲学家会先问"你确定是 500 而不是 502 吗"。甩锅侠的信条：你是 AI，你自己去看日志。

### Case 3: 开放讨论/创意（甩锅式讨论）
**场景**: 产品让想个活动名，顺手转嫁给 AI
**Prompt**:
```
618 活动要个名字，给我三个，定了就用。
```
**四维判定**:
- A/C: A（理由：没说活动主题、调性、品牌）
- M/V: M（理由：只要三个备选，不要解释）
- D/L: D（理由："定了就用"直接封死讨论空间）
- X/E: E（理由：要可直接拍板的成品）
- **最终代码**: AMDE
**为什么典型**: 这是甩锅侠版本的"讨论"——把选择权也一并甩出去。他不会像 AMLC 那样铺陈"我们这次想传达年轻化"，也不会像 ACDX 凌晨哲学家那样追问"名字的意义是什么"。甩锅侠的讨论 = 让 AI 出几个，他从里面挑一个盖章。

### Case 4: 失败/卡壳（继续甩）
**场景**: AI 给的代码跑不通，直接反手再甩
**Prompt**:
```
报错了，你自己看，搞定。
```
**四维判定**:
- A/C: A（理由：连报错内容都懒得贴）
- M/V: M（理由："你自己看"四字打包所有信息）
- D/L: D（理由："搞定"是终点指令，无讨价空间）
- X/E: E（理由：要能跑的代码，不要解释原因）
- **最终代码**: AMDE
**为什么典型**: 失败时反而更甩，是甩锅侠的标志性反应。佛系甲方会说"那咱再调一下"带入协作感，哲学家会反思"为什么会失败"。甩锅侠的信念：错是你错的，你修。他把自己定位成验收方，永远不下场。

### Case 5: 复杂多轮对话
**场景**: 做一个新功能，三轮都在甩
**Prompt**:
```
【轮1】给用户加个积分系统，你设计。
【轮2】看着办，能跑就行。
【轮3】上线吧，出问题你负责。
```
**四维判定**:
- A/C: A（理由：三轮累计零上下文，连业务规则都不给）
- M/V: M（理由：每轮都压到极简，不补充）
- D/L: D（理由："你设计""上线吧"全是命令，无讨论）
- X/E: E（理由：要闭环交付，不要方案对比）
- **最终代码**: AMDE
**为什么典型**: 多轮场景最能暴露人格稳定性。真甩锅侠三轮不破功，每一轮都在加码甩——从"你设计"到"能跑就行"到"你负责"，锅越甩越大。对比 AMLE 会在第二轮加一句"辛苦啦"，AMDX 会在第三轮开始反问"积分的哲学意义"。甩锅侠一路甩到底，还顺手把责任也甩了。

# AIBTI 用户旅程故事板

> 评审视角：一个真实的 Claude Code 重度用户，技术背景、略带玩世不恭、被过去的"星座 App"伤过。
> 核心原则：**LLM 语义分析** + **大神理论背书** + **不冤枉人** + **样本不足时温柔**。

---

## 阶段 1：发现（Discovery）

**场景**：周三晚 23:47，用户在 V2EX 技术节点刷到一个帖子 `"一个分析你 Claude Code 提示词的 MBTI 插件 aibti"`，带 GitHub 链接。点进去。

### 用户心理全貌
- **第一反应（3 秒内）**：看 Star 数、看最近 commit 时间、看 README 第一屏。"是不是又一个水项目？"
- **警觉点**：提示词是私密的——有人写过骂老板的、写过情绪崩溃的、写过公司代码。**只要一闻到"上传云端分析"就立刻关掉。**
- **期待点**：技术人爱玩分类，爱把自己贴标签。但前提是**这个标签有理论依据**，不是随机 16 选 1。

### 关键触点（必须在 README 第一屏解决）
1. **标题一句话**：`Claude Code 的人格 X 光：LLM 读你两百条 prompt，告诉你你是 16 型中的谁（本地跑，不上传）`
2. **一张动图 GIF**：终端里 `/aibti-scan` 滚动，最后定格在一张 ASCII 人格卡，主人格 + 辅人格 + 一句话诊断。
3. **三行信任状**：
   - `✓ 100% 本地，只读你的 ~/.claude/projects/*.jsonl`
   - `✓ 开源，代码全公开，任何 prompt 不出机器`
   - `✓ 用大神 X/Y/Z 的人格理论做底层（非星座，非塔罗）`
4. **一句反预期**：`不会冤枉你。样本不够时我们会告诉你"再用几天"，不会瞎猜。`

### 潜在质疑 → 回应脚本
| 质疑 | 一句回应 |
|---|---|
| 又一个星座玩具？ | 底层是 [大神理论] + LLM 语义；16 型的依据全部在 docs/theory.md |
| MBTI 早被证伪了吧？ | 我们不是心理学人格，是**"你如何使唤 AI"**的行为画像，类比学习。|
| 隐私怎么办？ | 源码里 grep 一下 `http` 就知道全程没网络请求，只有本地 LLM 调用。|
| 我懒得装 | `claude plugin install aibti`，30 秒，不信你试试。|

## 阶段 2：首次使用（First Use）—— 样本不足的黑暗时刻

**场景**：用户刚 install，运行 `/aibti-scan`。本机 Claude Code 只用了 3 天，共 5 条对话。

### 最致命的陷阱
**绝对不能出现的文案**：
- ❌ "样本不足，分析失败"
- ❌ "请积累更多数据后重试"（太客服）
- ❌ 直接给一个看似完整的报告（这等于冤枉新用户）

### 正确的体验设计（样本 < 20 时）
```
/aibti-scan

  扫描完成：发现 5 条对话（约 800 字提示词）

  ⚠ 现在下结论太早——就像听你说 3 句话就判你是什么人，不负责任。

  初步气质速写（仅供参考，置信度 18%）：
  ├─ 你倾向于： 给明确指令、期待直接结果
  └─ 你较少： 开放式探索、来回讨论

  建议：再用 Claude Code 7-10 天（目标 100+ 条对话），回来跑：
    /aibti-scan

  我会等你。你的数据越多，我越敢说真话。
```

### 为什么这样写
1. **坦诚置信度**（18%）——技术人吃这一套，比虚张声势强 10 倍。
2. **"听 3 句话就判人"比喻**——让用户理解"不是你的错，是样本问题"。
3. **给可执行的时间目标**（7-10 天 / 100 条）——比"稍后再试"有用。
4. **"我会等你"**——拟人但不谄媚，留下情感钩子，让用户一周后真的想回来。

### 阶段 2 副产物：埋一个回流钩子
- 装完 aibti 之后，在 Claude Code 的启动横幅偶尔提示一次：`你的 AIBTI 样本：47/100，再积累 53 条就能出首份报告`——**进度条比广告有效**。

## 阶段 3：第一份真实报告 —— 情绪的爆点

**场景**：两周后，用户累积 187 条对话，重跑 `/aibti-scan`。

### 看到主人格那一刻（0-2 秒）
用户盯着屏幕。**这一瞬间，产品生死在此。**
- 如果是个正面标签（例：`ARCH - 架构师型`）→ 截图发群。
- 如果是个**负面感受的标签**（例：`AMDE - 甩锅侠`）→ 沉默、关掉、删插件。

### 评审员亲历复盘：我自己被判为"甩锅侠"的那次
那次我真的生气了。不是因为**判得不对**，是因为**判得太冷**。报告说："您倾向于把复杂问题整块丢给 AI，缺乏拆解能力。"——像被 HR 绩效谈话。

**这种伤害来自三个源头**：
1. 标签名本身带贬义（"甩锅侠" ≠ "高效委派型"）
2. 诊断文案用"缺乏 X"而不是"擅长 Y"
3. 没有给出**正向重构视角**

### 正确的主人格页设计
```
┌───────────────────────────────────────────────┐
│  AMDE · 高效委派型（Assign-Merge-Delegate）   │
│  ────────────────────────────────────────     │
│  你的超能力：                                 │
│    把复杂问题一句话说清，交给 AI，拿结果。    │
│    这是**资深工程师和技术 Leader 的典型模式**。│
│                                               │
│  你要小心的暗面：                             │
│    当 AI 卡住时，你倾向于换一个 prompt 再丢， │
│    而不是拆解——有时候拆解 5 分钟 比重试 5 次快。│
│                                               │
│  一句话：你是在用"管理者思维"用 AI，这不是坏事。│
└───────────────────────────────────────────────┘

 理论出处：[大神 X《委派型认知》第 4 章]
 置信度：82%  样本：187 条  覆盖周期：14 天
```

### 温度的配方（三条铁律）
1. **先夸超能力，再提暗面**（比例 2:1）
2. **永远有大神理论出处**——让标签不像"系统乱贴"
3. **"一句话"收尾**——用户截图晒图时，只截这一行

## 阶段 4：反复复测 —— 进化轨迹与上瘾设计

**场景**：用户一个月后第三次跑 `/aibti-scan`。

### 进化视图（必做）
```
你的 AIBTI 漂移轨迹：

  Day 14   AMDE · 高效委派型      ████████░░  82%
  Day 30   AMDE → ARCH 混合态    ███████░░░  71% / 64%
  Day 60   ARCH · 架构师型        ██████████  88%

  变化摘要：
   你开始更频繁地在 prompt 里写"先列步骤，再写代码"。
   这是**从执行者到设计者的认知迁移**。漂亮的进化。
```

### 关键设计要点
1. **漂移不是退化**——哪怕从 ARCH 掉到 AMDE，也要写"你正在尝试更快迭代，这是切换期的典型表现"。
2. **两次报告之间必须显示差异**——不能让用户感觉"跟上次没区别"。
3. **每次跑完给一个"下周观察点"**：例 "下周看你会不会更多用 // 反问 AI 澄清——那是 SAGE 人格的信号"。

### 上瘾 vs. 健康
- ✅ 合理频率：每 2 周 / 每 50 条新对话一次
- ❌ 反模式：每天跑、每小时跑——**主动在短期重复跑时提示**："样本没变化，报告也不会变化。去多写点 prompt 再回来。"
- 这一点比微博/星座 App 高级——**我们不贩卖焦虑**。

## 阶段 5：分享传播 —— 隐私与社交的平衡

**场景**：用户拿到 ARCH 报告，手痒想发朋友圈 / 技术群。

### 导出方案（三档）
1. **一张卡片 PNG**（默认推荐）——只含人格名、一句话、置信度、AIBTI logo。**完全不含任何 prompt 片段。**
2. **Markdown 报告**（技术群适用）——含 top 5 行为证据，但**证据必须是聚合后的抽象描述**（"你 42% 的 prompt 以动词开头" 而非原句）。
3. **原始 JSON 导出**（本地调试/二次分析）——给 geek，明确标红"含原始 prompt 片段，请勿外传"。

### 隐私三条红线
1. **真实 prompt 永远不进入分享物**——哪怕用户主动想贴，默认导出都剥离。
2. **项目名/文件路径脱敏**——`~/work/alibaba-xxx/` 必须变成 `~/project_A/`。
3. **分享前弹一次确认**：`你即将导出卡片（不含任何对话内容），是否继续？`——让用户确认"知道自己在分享什么"。

### 社交扩散钩子
- 卡片底部一个小二维码 → `github.com/user/aibti`
- 一句 `我是 ARCH，你呢？` 生成默认 caption
- **不做排行榜**（排行榜=焦虑=退坑）

# ⚠️ 改进建议清单（可执行）

1. **[阶段 2 · 关键]** 样本 < 20 时，**严禁**出现 "失败/错误/不足" 字眼，改为"现在下结论太早"+ 明确时间目标（7-10 天 / 100 条）+ 置信度数字。

2. **[阶段 3 · 关键]** 所有 16 型标签都必须有**正向名称**（`AMDE=高效委派型`，而非"甩锅侠"）。"甩锅侠" 这类梗可以作为 easter egg 彩蛋在二级页露出，但主人格页一律正向命名。

3. **[阶段 3]** 主人格页文案模板强制执行 `超能力段（2）→ 暗面段（1）→ 一句话收尾`，比例 2:1，夸在前面。

4. **[阶段 3]** 每个人格标签必须带**大神理论出处**（`理论出处：[大神 X《XX》第 Y 章]`），否则用户会质疑"凭什么"。

5. **[阶段 1&3]** README 和报告里都要显式写 **"本地运行、零上传"**，且在报告页脚附一句 `本次扫描网络请求数：0`——用事实，而非承诺。

6. **[阶段 4]** 强制显示**漂移轨迹图**——两次报告间如果一模一样，要主动提示"样本变化不足，建议再积累 XX 条"，而不是出一份重复报告。

7. **[阶段 4 · 反上瘾]** 检测到用户在 24h 内重复跑 `/aibti-scan` 超过 2 次时，返回："你上次跑是 3 小时前，样本只新增了 2 条。报告不会变。先去写 prompt。"

8. **[阶段 5 · 隐私]** 默认导出的 PNG **绝对不含任何 prompt 原文**——哪怕用户在 `--verbose` 下也要二次确认。证据描述一律聚合化（"42% 以动词开头"而非引用具体句子）。

9. **[阶段 5]** 项目名/路径默认脱敏（`project_A/B/C`），用户需显式 `--no-redact` 才能保留——**默认安全**。

10. **[全局 · 温度]** 整个产品禁用以下词汇：`失败`、`错误`、`缺乏`、`不够`、`低`。替换为：`还太早`、`下次再看`、`更擅长 X`、`样本待积累`、`正在形成`。**AIBTI 的基调是"等你慢慢长大"，不是"给你下诊断"。**
